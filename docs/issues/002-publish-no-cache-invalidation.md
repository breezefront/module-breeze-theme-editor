# Issue 002: Published changes not visible on storefront without manual cache flush

**Severity:** Critical  
**Area:** `etc/frontend/di.xml`, `Plugin/Mutation/InvalidateCacheAfterMutation.php`  
**Type:** Bug  
**Status:** In progress — partially addressed in `c7c4abe`, two root causes remain

---

## Problem

After clicking **Publish** in the Theme Editor, changes are not visible on the
storefront until the admin manually clears the Magento cache
(`bin/magento cache:flush` or Admin > System > Cache Management > Flush).

---

## History

### Fix attempt — `c7c4abe` (2026-01-13)

The plugin was attached to a non-existent class `PublishDraft` instead of
`Publish`. This was corrected:

```xml
<!-- BEFORE -->
<type name="Swissup\BreezeThemeEditor\Model\Service\PublishDraft">

<!-- AFTER -->
<type name="Swissup\BreezeThemeEditor\Model\Service\Publish">
```

Later, the architecture was refactored: cache invalidation moved from the
service layer to the GraphQL resolver layer (`Model/Resolver/Mutation/*`).
The plugin registrations were updated to point at resolver classes, but were
left in `etc/frontend/di.xml` — which introduced the two root causes below.

---

## Root Cause 1 — Critical: plugin registered in wrong DI scope

Magento has a dedicated `graphql` application area. GraphQL requests are
dispatched through `Magento\GraphQl\Controller\GraphQl`, which boots the
`graphql` area, **not** `frontend`. Area-scoped DI configs are loaded
exclusively for their own area:

```
etc/di.xml           ← global, always loaded
etc/frontend/di.xml  ← loaded only in "frontend" area
etc/graphql/di.xml   ← loaded only in "graphql" area
```

All three cache-invalidation plugin registrations are in `etc/frontend/di.xml`:

```xml
<type name="Swissup\BreezeThemeEditor\Model\Resolver\Mutation\Publish">
    <plugin name="swissup_bte_invalidate_cache_after_publish" ... />
</type>
```

When the Theme Editor calls `Publish`, `SaveValues`, or `Rollback` via
GraphQL, the DI container for the `graphql` area is active. The `frontend`
plugins are invisible in that context — **`afterResolve` never executes**.

---

## Root Cause 2 — High: Full Page Cache (FPC) is never invalidated

`InvalidateCacheAfterMutation::invalidateCache()` calls:

```php
$this->cache->clean(['bte_theme_variables']);
```

This cleans only the BTE block-level cache. Magento's **Full Page Cache**
(`full_page`) wraps entire page responses, including the `<style>` block
generated by `Block/ThemeCssVariables.php`. Cleaning the block cache does
not evict the FPC-cached pages.

Even if Root Cause 1 is fixed, FPC-cached pages will continue to serve the
old CSS variables until FPC expires or is flushed manually.

---

## Root Cause 3 — Low: dead branch in `isSuccessful()`

```php
// Plugin/Mutation/InvalidateCacheAfterMutation.php:61
// This branch never matches — Publish::resolve() returns a flat array:
// ['success' => true, 'message' => ..., 'publication' => [...]]
if (isset($result['publishBreezeThemeEditor']['success'])) {
    return (bool)$result['publishBreezeThemeEditor']['success'];
}
```

The first branch (`$result['success']`) correctly handles all three resolvers,
so the bug is silent. The misleading comment and dead code should be removed.

---

## Fix Plan

### Step 1 — Move plugin registrations to `etc/graphql/di.xml`

Move (or copy) all three `<type>` blocks from `etc/frontend/di.xml` to
`etc/graphql/di.xml`:

```xml
<!-- etc/graphql/di.xml -->
<type name="Swissup\BreezeThemeEditor\Model\Resolver\Mutation\SaveValues">
    <plugin name="swissup_bte_invalidate_cache_after_save"
            type="Swissup\BreezeThemeEditor\Plugin\Mutation\InvalidateCacheAfterMutation"
            sortOrder="100"/>
</type>

<type name="Swissup\BreezeThemeEditor\Model\Resolver\Mutation\Publish">
    <plugin name="swissup_bte_invalidate_cache_after_publish"
            type="Swissup\BreezeThemeEditor\Plugin\Mutation\InvalidateCacheAfterMutation"
            sortOrder="100"/>
</type>

<type name="Swissup\BreezeThemeEditor\Model\Resolver\Mutation\Rollback">
    <plugin name="swissup_bte_invalidate_cache_after_rollback"
            type="Swissup\BreezeThemeEditor\Plugin\Mutation\InvalidateCacheAfterMutation"
            sortOrder="100"/>
</type>
```

Remove the same blocks from `etc/frontend/di.xml`.

### Step 2 — Add FPC invalidation to the plugin

Inject `TypeListInterface` and invalidate `full_page` alongside the block cache:

```php
use Magento\Framework\App\Cache\TypeListInterface;

public function __construct(
    private CacheInterface $cache,
    private TypeListInterface $cacheTypeList,
    private LoggerInterface $logger
) {}

private function invalidateCache(): void
{
    $this->cache->clean(['bte_theme_variables']);
    $this->cacheTypeList->invalidate('full_page');
}
```

### Step 3 — Clean up `isSuccessful()`

Remove the dead branch and the incorrect comment:

```php
private function isSuccessful($result): bool
{
    if (!is_array($result)) {
        return false;
    }

    return isset($result['success']) && (bool)$result['success'];
}
```

## How to Test

1. Enable Full Page Cache (`bin/magento cache:enable full_page`)
2. Open the storefront — note the current colors/fonts
3. Open Theme Editor, change a color, click **Save Draft**
4. Verify storefront is unchanged (draft only)
5. Click **Publish**
6. Reload the storefront **without** any manual cache flush
7. Changes must appear immediately

Repeat steps 3–7 for **Rollback** to verify that rollback also triggers
cache invalidation.
